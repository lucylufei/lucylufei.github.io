CUDA Notes

Global functions are also called "kernels". It's the functions that you may call from the host side using CUDA kernel call semantics (<<<...>>>).

Device functions can only be called from other device or global functions. __device__ functions cannot be called from host code

The main() function runs on the CPU so it cannot call a __device__ function that runs on the GPU.

General CUDA + Vulkan Information

For example:

  • CPU sets up the BVH tree

  • Sends a shader that's bound to a particular BVH tree, and the GPU will use the shader and the BVH tree to compute the results

    • This call is probably the traceNV()
    • This is the API

  • The GPU has ISA that runs the traceNV() functionality

  • For example, cuda might send a kernel that's bound to a page table so the GPU can use just the data that it needs to do something.

 

Documentation

  • For CUDA, it compiles down to PTX

  • Everything should compile down to SASS, which is the disassembly of the machine code.

  • Check the CUDA Binary Utilities guide to see what the possible opcodes are. These opcodes vary by architecture (i.e. Volta, Turing, etc.)

    • 💡 Check the CUDA 10 version and diff between architectures to see if there are any ray tracing machine ISA op codes

    • https://docs.nvidia.com/cuda/cuda-binary-utilities/index.html#instruction-set-ref

    • 💡 Look into QSPC instruction

      https://docs.nvidia.com/gameworks/content/gameworkslibrary/physx/guide/Manual/SceneQueries.html#scenequeries

Emerald Meeting - June 12, 2020

  • https://gem5-gpu.cs.wisc.edu/wiki/benchmarks

    Instructions for how to compile and run the gem5-gpu benchmarks

  • Theoretically, all the CUDA capabilities in gem5-gpu are still included in Emerald

  • Process

    1. Run CUDA on gem5-gpu
    2. gem5 runs CPU portion of the code
    3. invokes gpgpu-sim when it hits magic CUDA calls
    4. proceeds in gpgpu-sim

  • Running Francois's ambient occlusion benchmark doesn't require Emerald. This is purely CUDA and can be run on regular gpgpu-sim

    • The goal of running this on Emerald is to eventually be able to consider hybrid rendering

  • Emerald is gem5-gpu's gpgpu-sim with the gem5 memory model + the addition of the graphics pipeline.

    • Emerald is the only version with graphics pipeline
    • gem5-gpu has a different memory model because the CPU portions of the simulation are also simulated. (Regular gpgpu-sim runs the CPU code on the actual CPU and only invokes the simulator for GPU workloads)

  • Emerald's standalone mode is actually quite useless to us for now. This mode only takes in OpenGL traces.

    • RT API is likely to be added to CUDA
    • Potentially add some Vulkan support to gpgpu-sim
    • Only case where this may be useful is if we use Mesa to compile Vulkan and then run in Emerald using the TGSI-PTX translation

Random Notes:

  • JIT (Just in Time) compiler compiles as the code is running.

    • This is better than an interpreter because it runs compiled code.
    • This is better than pre-compiling because it can catch errors and help in the debugging process.
    • It is also platform adaptable; it's not compiled to any particular architecture.

  • P vs. NP (Polynomial vs. Non-Deterministic Polynomial)

    • P problems can be solved in some polynomial time (not exponential)
    • NP problems can be checked in some polynomial time (but don't know how long to solve)
    • NP-Complete are the hardest NP problems
    • NP-Hard are exponentially hard problems
